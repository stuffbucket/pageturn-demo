# Diagnostic 2026-05-14: bleed-through, settle asymmetry, fiducial-grid stretch sweep

> Status: **Diagnosis only.** No fixes are applied here. Tracking issues
> are filed for each distinct root-cause hypothesis; the fixes get
> separate PRs informed by this report.

This report consolidates a focused diagnostic pass driven by three
defects the user is still observing on the latest `main`
(commit `265799f`):

1. **Bleed-through during lift** &mdash; the next leaf shows through the
   front of the turning page (Refs #54 #58).
2. **Settle asymmetry** &mdash; pages settling left smooth out with a
   transient curl near the spine; pages settling right "snap" flat early.
3. **Stretching away from the crease** &mdash; FR-P1 inextensibility is
   visibly violated for some drag origins even though the unit-test
   invariant passes for synthetic inputs (Refs #18 #50).

Companion deliverables (this PR):

  mirror of the harness's analytic fiducial position function, exposed to

- New telemetry events: `fiducial-positions`, `surface-area-sample`,
  `pre-settle-state`, `post-settle-state`, `drag-origin-fiducial`.
- Live FR-P1 area-ratio readout in the debug HUD (color-banded
  green/yellow/red at 1% / 5% thresholds), plus a "drag origin fiducial"

- Long-press capture sidecar now embeds the live 5x7 fiducial grid +

- 280 deterministic harness scenarios under
  `harness/scenarios/fiducial-grid-sweep/` (one per page, fiducial,

## Hypotheses (ranked by evidence)

### H1 (high confidence): originY snap quantization &rarr; per-row stretch (Refs #18 #50)

**Evidence pointer:** 5 of 8 user captures hit `originY = ±0.700`
exactly; the analytic heat-map (`docs/diagnostic-heatmap-2026-05-14.html`)
shows a 2.28% peak FR-P1 deviation in the rest-frame model that comes
entirely from the *bend envelope* `b * t * sin(2 * uAngle)`, but the
LIVE shader has an additional originY-snap step inside
`creaseFromDrag`. PR #59 fixed an originY-deviation area-growth bug in
the developable shader; the user's captures are on `?dev-surface=1` and
still show stretch, so the fix is incomplete.

**Predicted heat-map signature (when run against the live GPU):**
patches at the column boundary (i=0 vs i=4) and at the row clamp
boundary (j=0 vs j=6) where the originY snap quantization changes the
crease parameterization in a way the analytic model cannot see.

**Repro:** drag from fiducial (i=0, j=3) along +x &mdash; the snap that
moves originY off the spine cell will produce a different curl than the
same drag from (i=0, j=2). The diagnostic HUD's new "origin fid" line
makes this single-glance observable.

### H2 (high confidence): aero settle bend-envelope sign asymmetry across the spine

**Evidence pointer:** `src/book/SettlePhysics.ts:135` &ndash; the integrator
runs `phi` in `[0, π]` with `dir * G * sin(phi)`. The shader's bend
envelope (`Book.ts` FLIP_VERT inline shader) is
`uBendAmount * t * sin(2 * uDihedral)`. For a forward commit settling to
`phi = π`, the shader passes through `phi ≈ 3π/4` where `sin(2 * 3π/4) = -1`
&mdash; the curl points one way. For a reverse commit settling to
`phi = 0`, the shader passes through `phi ≈ π/4` where `sin(2 * π/4) = +1` &mdash;
the curl points the *other* way. The aerodynamic `b`-puff is added on
top with the same magnitude (`κ * φ̇²`, sign-independent), so during a
forward settle the puff *visually augments* the natural bend, while
during a reverse settle it *visually opposes* the bend. The result the
user described &mdash; "left smooths with a transient curl, right snaps
flat" &mdash; matches this prediction direction-for-direction.

**Why the unit tests don't catch it:** `SettlePhysics.test.ts` asserts
on `phi` and `b` symmetrically; the asymmetry only exists when those
values are *consumed by the FLIP_VERT shader*. The new `pre-settle-state`
+ `post-settle-state` telemetry events let the harness diff observed vs.
mirrored trajectories without going through GLSL.

### H3 (medium confidence): polygon-offset is insufficient for the curled back face when uDihedral is near &pi;/2 (Refs #31 #54 #58)

**Evidence pointer:** `Book.ts` configures the turning-page material as
`DoubleSide` with `polygonOffsetFactor = -2, polygonOffsetUnits = -2`,
and the fragment shader uses `gl_FrontFacing` to switch textures. Three
publicly tracked Three.js issues confirm this is fragile:

- mrdoob/three.js#32827 &mdash; `polygonOffset` does not respect the
  depth coordinate system uniformly across drivers.
- mrdoob/three.js#25149 &mdash; `gl_FrontFacing` fails when a
  `DoubleSide` object is also marked transparent (we are not transparent
  yet, but the disabled popup feature needed it; reactivating ADR-0001
  will reintroduce the failure mode).
- mrdoob/three.js#20625 &mdash; `DoubleSide` plus polygon offset can
  flip the visible face on Adreno-class GPUs.

**Why the captures point at this:** capture 3's `dihedral = 1.725` is
just past π/2 &mdash; the regime where the back face becomes
parallel to the camera and the polygon-offset bias has the smallest
effect. Capture 4 (`dihedral = 1.136`) and capture 5 (`dihedral = 2.138`)
straddle that regime; the user reported these as the worst bleed-through
moments.

### H4 (low confidence): drag-out-of-window clamp &rarr; degenerate originY at `dragPoint.x = -W` (Refs #58)

**Evidence pointer:** captures 2 and 7 have `dragPoint.x ≈ -1.001`
(just past the negative page-width boundary) with `dragProgress = 0`,
both from `j = -1` (book closed). The shader is presenting a degenerate
state and the user described this as "snap flat" on reverse turns. The
existing `bleed-through-mid-fold` and `drag-out-of-window` scenarios may
not exercise the clamp at exactly this corner; the new
`fiducial-grid-sweep/fid-l-0-3-mx.json` does.

## Prior-art research

- **iBooks-style conical page curl** &mdash; the canonical implementation
  is described in
  [Implementing iBooks page curling using a conical deformation algorithm](http://wdnuon.blogspot.com/2010/05/implementing-ibooks-page-curling-using.html).
  iBooks uses a *conical* (not cylindrical) developable surface; the
  area-preserving cone is what keeps FR-P1 satisfied at all dihedrals.
  Our cylindrical-curl model in `DevelopableSurface.ts` is a v1
  simplification (see PRD-page-model.md Q1) and is the geometric reason
  H1 still bites us.
- **Computer graphics page-turning survey** &mdash; the
  [Springer survey by Hong et al.](https://link.springer.com/article/10.1007/s00799-009-0055-3)
  summarizes the Beier&ndash;Neely-style page curl, the conical-surface
  family, and FEM cloth approaches; all of them treat the spine clamp
  as an *additional* boundary condition in the bend equation rather than
  a UV-space mask, which matches our `pos.x > spineEps` `tanh` clamp in
  FLIP_VERT. The asymmetry mode H2 is well-known in the cloth literature
  &mdash; see Provot 1995 mass-spring papers cited in
  `SettlePhysics.ts`.
- **gl-transitions InvertedPageCurl** &mdash; the
  [reference shader](https://github.com/gl-transitions/gl-transitions/blob/master/transitions/InvertedPageCurl.glsl)
  hard-codes `discard` on the back face under certain conditions to
  avoid bleed-through; we do not, which is the nearest publicly visible
  precedent for H3.
- **Cyanilux spellbook breakdown** &mdash; the
  [tutorial](https://www.cyanilux.com/tutorials/spellbook-breakdown/)
  documents that double-sided book pages "bleed" without explicit depth
  bias on the back-face render; their fix is a separate render pass
  rather than `polygonOffset`. Treating that as a starting reference for
  the H3 fix.
- **Chris Luke "anatomy of a page curl"** &mdash;
  [blog](https://blog.flirble.org/2010/10/08/the-anatomy-of-a-page-curl/)
  describes the same originY-snap behaviour (under the name "anchor
  point quantization") in a Cocoa Touch implementation; the proposed
  workaround is to interpolate originY linearly across the snap zone
  rather than clamp it. Direct precedent for an H1 fix.
- **turn.js issue #157** &mdash;
  [Blurry page when corner peels and 1 pixel vertical center flickering](https://github.com/blasten/turn.js/issues/157)
  is the closest community-tracked analog to our visible bleed-through;
  the fix in turn.js was to nudge the page out of the spine cell by
  `0.5px` in CSS, which is conceptually similar to our polygon offset.

## Recommended fix order

1. **H4** (drag-clamp): cheapest, isolated to `main.ts` pointer math, no
   shader changes. Likely a one-line guard.
2. **H2** (settle asymmetry): change `setBendAmount` to take a `dir`
   argument so the renderer can choose the puff sign. Localized to
   `SettlePhysics.ts` + `Book.ts`. Add a regression test in
   `Settle.test.ts` that asserts puff-sign symmetry under spine-mirror.
3. **H3** (bleed-through): try the `discard` approach from
   gl-transitions before reaching for a separate-pass refactor. Bisect
   on `polygonOffsetUnits` from -2 down to -8 to find the smallest value
   that closes the gap.
4. **H1** (originY snap): biggest blast radius. Run the 280-scenario
   sweep against the live GPU (this PR ships the scenarios, the runner
   just needs a `--dir harness/scenarios/fiducial-grid-sweep` flag) to
   confirm the patch shape, then convert the snap to a smoothstep and
   re-baseline the trajectory tests under `harness/baselines/`.
